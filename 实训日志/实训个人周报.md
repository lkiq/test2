## 项目实践过程遇到的问题以及解决问题时的思考过程和方法

### 问题1：需求评审中发现"学习进度展示"功能缺失
**现象**：评审时指出学生端缺少学习进度闭环，无法展示学习成果
**思考过程**：
- 是否需要新增独立的数据表存储进度？
- 进度数据是否可以由现有表聚合生成？
**解决方法**：
- 采用数据聚合策略，避免新增复杂学习事件表
- 进度由learning_task + assessment_result + interview_record + resume_analysis聚合生成
- 新增3个进度查询接口（GET /api/student/progress/overview/skills/report）
- 成长报告复用recommendation_record表（type=GROWTH_REPORT）
**收获**：数据聚合比新增表更轻量，降低系统复杂度，同时满足功能需求

### 问题2：AI调用缺乏审计机制
**现象**：评审时指出AI服务不可追踪，无法排查问题和优化成本
**思考过程**：
- 需要记录哪些信息？（调用场景、Prompt、响应、耗时、状态）
- 如何平衡隐私保护和可追溯性？（Prompt存摘要而非全文）
**解决方法**：
- 新增ai_call_log表，记录：user_id、scene、prompt_summary、response_source、status、duration_ms、fallback_reason
- 明确AI调用链路：Prompt模板 → 参数组装 → API调用 → JSON解析 → 缓存 → 降级 → 记录日志
- 所有降级方案均记录状态（status=FALLBACK，fallback_reason注明原因）
**收获**：审计日志设计要兼顾隐私和可追溯，摘要+状态+耗时是关键字段

### 问题3：项目案例中功能超出V1.0范围
**现象**：项目案例包含16项功能，但需求规格说明书V1.0仅定义14项
**思考过程**：
- 是否需要在当前版本中实现所有案例功能？
- 如何避免"案例功能未体现"的评审风险？
**解决方法**：
- 明确本次完善只按V1.0实现范围处理，不把案例16项全部纳入
- 新增第6章「后续迭代清单」，将R-015~R-018及高校后台、完整企业招聘流程、语音客服、外部招聘数据接入等明确列为V2.0
- 当前设计预留最小扩展点（字段或菜单入口），确保V2.0无需重大重构
**收获**：架构设计要区分当前交付和后续迭代，预留扩展点比过度设计更重要

### 问题4：Git提交时中文文件名显示异常
**现象**：git status显示中文文件名为转义编码（\347\263\273\347\273\237）
**思考过程**：
- Git默认对非ASCII字符进行转义显示
- 需要配置Git正确显示中文文件名
**解决方法**：
- 执行git config --global core.quotepath false
- 重新添加文件后正常显示中文文件名
**收获**：跨平台开发要注意编码配置，Git的quotepath设置影响中文文件名显示

## 项目实践过程遇到的问题以及解决问题时的思考过程和方法

### 问题1：测试环境数据库依赖导致用例无法独立运行
**现象**：JUnit测试依赖外部MySQL，换机器即失败，无法在CI环境执行。
**思考过程**：@SpringBootTest默认加载dev配置导致强依赖外部数据库；能否用内存数据库替代？H2对MyBatis-Plus语法兼容性如何？
**解决方法**：引入H2内存数据库替代MySQL，@ActiveProfiles("dev")加载配置，H2自动建表；DeepSeek外部调用用MockWebServer模拟，隔离真实API。
**收获**：单元/集成测试要追求环境无关性，内存数据库+Mock是隔离外部依赖的标准做法。

### 问题2：MockMvc测试中JSON响应字段断言困难
**现象**：接口返回JSON嵌套较深，jsonPath断言易写错路径（如$.data.token与$.data.user.token混淆）。
**思考过程**：jsonPath路径表达式准确性难保证；能否反序列化为对象断言？统一响应结构Result<T>如何便捷处理？
**解决方法**：用ObjectMapper将响应反序列化为Map或DTO，对code/message/data核心字段分层断言，封装通用断言工具减少重复代码。
**收获**：JSON断言要结合jsonPath（简单字段）与对象反序列化（复杂结构），兼顾可读性与稳定性。

### 问题3：API Key硬编码的安全风险排查
**现象**：QA初评指出启动后端.bat中明文写入DeepSeek API Key，存在密钥泄露风险。
**思考过程**：硬编码密钥有哪些风险（仓库泄露即密钥泄露、无法区分环境、审计困难）？密钥应存储在哪里？如何兼顾便捷性与安全性？
**解决方法**：启动脚本从.env文件加载环境变量并加入.gitignore；application-dev.yml用${DEEPSEEK_API_KEY:默认值}占位符；新增.env.example模板供团队参考。
**收获**：密钥管理要遵循"配置与代码分离"原则，.env.example+环境变量是中小项目最佳实践。

### 问题4：E2E验证清单的粒度把握
**现象**：初版E2E用例要么过粗（一条覆盖多模块）要么过细（每按钮一条），难以执行。
**思考过程**：E2E目的是验证业务闭环还是覆盖分支？用例数量如何平衡覆盖度与维护成本？如何让非技术人员也能执行？
**解决方法**：以"业务场景"为单位设计用例，每条含编号/场景/前置条件/操作步骤/期望结果/状态；56项用例覆盖13模块，平均每模块4-5项。
**收获**：E2E用例要以业务场景为导向，操作步骤要具体到非技术人员可执行的程度。
