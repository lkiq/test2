# 注册优化、密码找回与HR企业信息 — 可行性评估报告

> **评估日期**：2026-06-28  
> **项目**：IT求职平台（Spring Boot 3 + Vue 3 + Element Plus）  
> **评估方式**：多维度打分机制（满分 100 分，≥70 分视为可行）

---

## 一、需求概述

| 编号 | 需求 | 描述 |

## 四、R2：邮箱找回密码

### 4.1 详细方案

```
注册时（R1改造后）    邮箱必填且唯一
       ↓
忘记密码页面          输入注册邮箱
       ↓
后端生成6位验证码    存入 Redis（Key: reset:code:{email}，TTL: 5分钟）
       ↓
发送邮件              通过 SMTP 发送验证码到用户邮箱
       ↓
用户输入验证码        校验 Redis 中的验证码
       ↓
设置新密码            BCrypt 加密后更新 user 表
       ↓
完成                 跳转登录页
```

### 4.2 变更范围

```
【新增依赖】pom.xml 添加：
  - spring-boot-starter-mail

【新增配置】application-dev.yml：
  - spring.mail.host（SMTP服务器）
  - spring.mail.port
  - spring.mail.username（发件邮箱）
  - spring.mail.password（邮箱授权码）
  - spring.mail.properties.mail.smtp.auth=true
  - spring.mail.properties.mail.smtp.starttls.enable=true

【新增代码】后端：
  - MailService.java — 邮件发送服务
  - PasswordController.java — POST /api/auth/forgot-password（发送验证码）
                              POST /api/auth/reset-password（验证+重置）
  - ForgotPasswordRequest.java / ResetPasswordRequest.java — DTO

【新增代码】前端：
  - ForgotPasswordView.vue — 忘记密码页面
  - src/api/auth.ts — 新增 forgotPassword() / resetPassword() API
  - router/index.ts — 新增 /forgot-password 路由
  - LoginView.vue — 添加"忘记密码？"链接
```

### 4.3 可行性评估

| 评估维度 | 得分 | 满分 | 说明 |
|----------|:---:|:---:|------|
| **技术依赖** | 13 | 20 | ⚠️ **核心瓶颈**：需要外部 SMTP 邮箱服务（如QQ邮箱/163邮箱/企业邮箱），需获取授权码，非纯代码可控 |
| **后端改动量** | 16 | 20 | 新增 1 个 Controller + 1 个 Service，改动量适中 |
| **前端改动量** | 15 | 20 | 新增 1 个页面 + 少量路由修改，工作量可控 |
| **安全性** | 14 | 15 | Redis TTL 5分钟防暴力破解 + 6位随机码 + 邮箱唯一性保障，方案较安全 |
| **Redis 基础设施** | 10 | 10 | ✅ Redis 已运行，验证码存储方案零额外成本 |
| **邮件服务可用性** | 8 | 15 | ⚠️ 依赖第三方 SMTP，可能受频率限制（QQ邮箱每日500封上限），高并发场景需升级企业邮箱或云邮件服务 |
| **总分** | **76** | 100 | 🟡 **可行但依赖外部服务** |

> **关键风险**：邮件服务依赖外部 SMTP，需提前申请授权码。建议使用 163 邮箱（限制较宽松）或接入阿里云/腾讯云邮件推送。

## 五、R3：HR填写公司信息

### 5.1 详细方案

#### 新增企业实体

```sql
CREATE TABLE enterprise (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    hr_user_id BIGINT NOT NULL COMMENT '关联的HR用户ID',
    company_name VARCHAR(100) NOT NULL COMMENT '公司名称',
    company_industry VARCHAR(50) COMMENT '所属行业',
    company_size VARCHAR(20) COMMENT '公司规模：1-50/51-200/201-500/501-1000/1000+',
    company_address VARCHAR(200) COMMENT '公司地址',
    company_description TEXT COMMENT '公司简介',
    contact_name VARCHAR(50) COMMENT '联系人姓名',
    contact_position VARCHAR(50) COMMENT '联系人职位',
    verify_status VARCHAR(20) DEFAULT 'PENDING' COMMENT '认证状态：PENDING/APPROVED/REJECTED',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_hr_user_id (hr_user_id),
    INDEX idx_verify_status (verify_status)
);
```

#### 注册流程改造（HR角色）

```
【当前流程】
选HR角色 → 填用户名密码 → 注册成功（直接可发布职位）

【改造后流程】
选HR角色 → 填用户名密码 + 公司信息 → 注册成功 → 公司状态=PENDING
                                                         ↓
                                           管理员审核 → APPROVED → 可发布职位
                                                     → REJECTED → 需重新提交
```

#### 变更范围

```
【数据库】
  - 新建 enterprise 表
  - user 表无结构变更（关联通过 hr_user_id）

【后端】
  - Enterprise.java — 实体类
  - EnterpriseMapper.java — MyBatis-Plus Mapper
  - EnterpriseService.java — 业务逻辑
  - RegisterRequest.java — HR角色时增加企业信息字段校验（条件校验）
  - AuthServiceImpl.java — HR注册时同时创建 enterprise 记录
  - AdminEnterpriseController.java — 管理员审核企业接口
  - enterprise_fields:
    - companyName（必填）
    - companyIndustry（必填）
    - companySize（选填）
    - companyAddress（选填）
    - contactName（必填）
    - contactPosition（选填）

【前端】
  - RegisterView.vue — HR角色时展示企业信息表单区域（v-if="form.role === 'HR'"）
  - EnterpriseHomeView.vue — HR首页增加"完善企业信息"入口
  - AdminEnterpriseView.vue — 管理员审核页面
```

### 5.2 可行性评估

| 评估维度 | 得分 | 满分 | 说明 |

| **数据库设计** | 18 | 20 | 新建 enterprise 表，与 user 表通过 hr_user_id 一对一关联，设计清晰 |
| **后端改动量** | 14 | 20 | 新增 Entity + Mapper + Service + Controller，RegisterRequest 条件校验有一定复杂度 |

## 九、风险与注意事项

| 风险 | 影响等级 | 应对措施 |
|------|:--:|------|
| **SMTP 邮件服务不稳定** | 🔴 高 | 优先使用 163 邮箱（限制较宽松），备注可升级腾讯云邮件推送 SES |
| **邮箱唯一性约束与历史数据冲突** | 🟡 中 | 上线前扫描 user 表，对空/重复邮箱做数据清洗脚本 |
| **HR企业信息审核增加运营成本** | 🟡 中 | 初期可设 auto-approve 模式，待数据量上来后再引入人工审核 |
| **注册表单过长降低转化率** | 🟢 低 | 保持分步注册设计，第一步选角色，第二步填信息（5-8个字段） |
| **user 表缺少 is_deleted 列** | 🟡 中 | 当前实体有 @TableLogic 但表无该列，建议在本次迁移中一并补充 |

### 综合可行性评分：**80 / 100** 🟢

三个需求在技术上均可行：

- **R1（注册增加个人信息）**：改动最小，数据库已有基础字段，82分，**强烈建议优先实施**。
