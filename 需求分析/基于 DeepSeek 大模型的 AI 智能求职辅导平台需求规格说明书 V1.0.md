## 摘要

本文档描述“基于 DeepSeek 大模型的 AI 智能求职辅导平台”的软件需求。系统面向大学生求职者、企业 HR 和平台管理员，重点解决学生职业方向不清晰、岗位能力差距难定位、学习路径不明确、简历与面试准备效率低，以及企业 HR 难以根据项目需求快速判断岗位与候选人的问题。

本版本采用“MVP 核心闭环 + 轻量亮点扩展”方案：学生端为 30 天核心交付，企业端保留 AI 项目需求解析和岗位/候选人推荐亮点，管理端和智能客服提供基础可演示能力。

# 1 简介

## 1.1 目的

本文档用于明确 AI 智能求职辅导平台 V1.0 的功能需求、非功能需求、接口需求、数据范围和质量要求，为后续概要设计、详细设计、开发、测试、验收提供依据。预期读者包括项目管理人员、需求分析人员、产品设计人员、开发人员、测试人员、指导老师及评审人员。

## 1.2 范围

本系统 V1.0 聚焦 30 天实训项目可完成范围，核心目标是打通学生端求职辅导闭环，并提供企业端 AI 推荐亮点演示。

包括内容：

- 学生端：注册登录、求职画像、职业能力测评、AI 职业方向探索、岗位匹配、能力差距分析、学习路径规划、简历优化、模拟面试、学习进度展示。
- 企业端：企业 HR 输入项目描述或岗位需求，系统解析岗位与技能要求，并推荐候选人。
- 管理端：用户管理、岗位技能词典维护、基础运营数据看板。
- 智能客服：基于文字的常见问题问答、岗位介绍和学习建议。

不包括内容：

- 移动 App、小程序端。
- 真实招聘平台数据接入。
- 语音客服、人工客服工单流转。
- 高校管理后台完整业务闭环。
- 企业端完整招聘流程，如复杂转化漏斗、批量邀约、录用管理。
- 多租户、复杂权限审批、商业支付结算。

# 2 总体概述

## 2.1 软件概述

### 2.1.1 项目介绍

2025 年全国高校毕业生人数突破 1200 万，就业竞争压力持续增大。大量学生在毕业前存在职业方向模糊、岗位技能要求不了解、学习路径不清晰、简历与面试准备不足等问题。互联网企业也面临候选人筛选效率低、关键词匹配失准、难以根据项目需求快速判断所需岗位的问题。

本项目基于 DeepSeek 大模型能力，建设面向学生和企业 HR 的 AI 智能求职辅导平台。学生端以“发现方向 → 定位差距 → 提升能力 → 准备求职”为主线，帮助学生完成职业定位和能力提升。企业端以“项目需求 → 岗位解析 → 候选人推荐”为亮点，帮助 HR 快速获得招聘参考。

### 2.1.2 产品环境介绍

系统采用 Web 架构，分为前端、后端、数据库、AI 服务四部分。

| 层级 | 技术选型 | 说明 |
|---|---|---|
| 前端 | Vue 3 + Vite + Element Plus + ECharts | 实现学生端、企业端、管理端页面与图表展示 |
| 后端 | Spring Boot 3 + MyBatis-Plus | 提供 REST API、业务逻辑和权限控制 |
| 数据库 | MySQL 8 | 存储用户、画像、岗位、技能、测评、推荐等数据 |
| 缓存 | Redis | 存储验证码、登录状态、AI 调用缓存、热点岗位数据 |
| AI 服务 | DeepSeek API + Prompt 模板 | 实现方向探索、简历优化、模拟面试、项目需求解析 |
| 部署 | Docker Compose | 单机部署，适合演示和实训交付 |

系统不强依赖复杂向量数据库。岗位技能词典、常见岗位 JD 和候选人画像采用 MySQL 结构化存储，必要时通过关键词召回、规则评分、DeepSeek 总结生成实现轻量 RAG 效果。

## 2.3 用户特征

| 用户类型 | 用户特征 | 使用目标 |
|---|---|---|
| 学生用户 | 大学生或应届生，对互联网岗位了解有限，有求职和学习提升需求 | 明确职业方向、补齐能力差距、优化简历与面试表现 |
| 企业 HR | 了解招聘流程，但未必熟悉项目技术栈和岗位技能细节 | 根据岗位或项目描述快速获得岗位建议和候选人推荐 |

## 2.4 假设和依赖关系

- 系统演示阶段使用预置岗位库、技能词典、模拟候选人数据，不接入真实招聘平台。
- AI 功能依赖 DeepSeek API；当 API 不可用时，系统使用本地规则、缓存结果和预置模板兜底。
- 用户上传简历仅用于本系统分析，需限制文件大小和类型。
- 团队规模为 3 人，开发周期约 30 天，功能设计以可演示、可验证、可扩展为原则。
- 系统主要面向桌面浏览器访问，不开发移动 App。

## 3.2 学生端核心模块

### 3.2.1 用户注册与求职画像

功能简介：学生注册后填写求职基础信息，系统生成个人求职画像，作为测评、推荐和学习规划的数据基础。

输入：账号、密码、手机号或邮箱；学历、学校、专业、年级；已掌握技能标签；目标岗位方向；期望城市、期望薪资、求职状态。

处理：校验账号唯一性和必填字段；将技能标签映射到系统技能词典；根据专业、技能、目标岗位生成画像摘要；当目标岗位为空时标记为“方向待探索”。

输出：用户基础信息、求职画像详情、标准化技能标签集合。

异常处理：重复账号提示重新输入；未识别技能允许作为自定义标签保存，并进入管理员待维护列表。

### 3.2.2 职业能力基线测评

功能简介：系统提供编程基础、逻辑推理、产品思维、技术理解、沟通协作五个维度的测评，生成学生能力基线。

输入：用户选择测评类型并提交题目答案。

处理：从题库中按维度抽取题目；计算每个维度得分和等级；判断优势维度和薄弱维度。MVP 阶段采用固定题库和简单难度分层，不实现复杂自适应算法。

输出：总分、维度分、能力等级、优势方向提示、AI 方向探索所需的结构化测评摘要。

异常处理：未完成测评时保存草稿；重复提交时以最后一次提交结果为准。

### 3.2.3 AI 职业方向探索与规划咨询

功能简介：当学生对职业方向不明确时，AI 根据画像、测评和对话补充信息，推荐 3-5 个互联网岗位方向并说明理由。

输入：求职画像、测评结果、学生在对话中补充的兴趣、偏好、价值取向。

处理：后端组装 Prompt，要求 DeepSeek 输出结构化 JSON；AI 结合本地岗位技能词典和岗位说明生成推荐；推荐结果包含岗位名称、匹配分、推荐理由、学习优先级。若 DeepSeek API 不可用，使用规则评分生成兜底推荐。

输出：3-5 个岗位方向、每个方向的推荐理由、初级到中级的能力成长建议。

异常处理：AI 返回格式错误时重新请求一次；二次失败时返回本地规则推荐结果。

### 3.2.4 岗位智能匹配与推荐

功能简介：系统根据学生画像、测评结果和已掌握技能，从岗位库中推荐适合岗位。

输入：学生技能标签、测评分数、目标城市；岗位库中的岗位 JD、技能要求、薪资范围。

处理：计算技能匹配分、测评适配分、城市匹配分；按加权分排序，参考权重为技能 50%、测评 30%、城市和薪资 20%；对岗位要求技能标记“已掌握、学习中、未掌握”。

输出：推荐岗位列表、匹配分、薪资范围、技能缺口、岗位详情和设置学习目标入口。

异常处理：无完全匹配岗位时返回相近岗位，并说明差距。

### 3.2.5 AI 岗位能力差距分析

功能简介：学生选择目标岗位后，系统对比岗位要求和学生当前能力，生成量化差距报告。

输入：目标岗位 ID、学生技能标签、测评结果、学习记录。

处理：查询岗位标准化技能要求；将学生能力按“了解、掌握、熟练、精通”四级映射；计算每项技能当前等级、目标等级和差距程度；使用 ECharts 展示能力雷达图和差距矩阵。

输出：能力差距报告、严重不足/需要提升/基本达标技能列表、学习路径生成入口。

异常处理：岗位缺少技能要求时提示管理员维护，并使用同类岗位模板兜底。

### 3.2.6 个性化学习路径规划

功能简介：系统根据差距报告生成“基础夯实 → 框架掌握 → 项目实战 → 面试冲刺”四阶段学习路径。

输入：能力差距报告、用户每日可学习时长、目标求职时间。

处理：按技能缺口严重程度排序；从学习资源库匹配知识点、练习题和项目任务；生成阶段计划和每日任务。

输出：学习路径总览、阶段目标、预计耗时、每日学习任务。

异常处理：用户未填写每日学习时长时默认每天 2 小时。

### 3.2.7 AI 简历智能优化

功能简介：学生上传简历，系统分析简历结构、关键词、项目经历和表达质量，给出修改建议和优化版本。

输入：PDF 或 DOCX 简历文件、目标岗位或目标 JD。

处理：后端提取简历文本；调用 DeepSeek 输出结构化分析结果；从完整性、岗位匹配、成果量化、专业表达四个维度评分；生成优化建议和示例改写。

输出：简历评分、问题清单、优化建议、面向目标岗位的简历片段改写。

异常处理：文件过大、格式不支持或文本提取失败时提示重新上传。

### 3.2.8 AI 模拟面试与表现评估

功能简介：系统根据目标岗位生成面试问题，学生回答后由 AI 追问并生成评估报告。

输入：目标岗位、面试类型、用户文字回答。

处理：根据岗位生成 5-8 道面试题；AI 根据回答进行最多 2 轮追问；面试结束后从逻辑性、专业度、沟通力、应变力、岗位匹配度五个维度评分。

输出：面试记录、五维度评分、回答亮点与改进建议、面试能力雷达图。

异常处理：AI 不可用时使用预置题库完成基础问答，评估报告改为规则模板。

## 3.3 企业端轻量亮点模块

### 3.3.1 HR 项目需求解析与候选人推荐

功能简介：企业 HR 输入项目描述或岗位需求，系统使用 DeepSeek 解析项目所需岗位、技能标签和人数建议，并从预置候选人库中推荐匹配学生。

输入：项目描述，如“开发一个在线商城，包含用户端、商家后台、推荐引擎、订单支付”；可选条件包括期望岗位、技能偏好、学历要求、城市。

处理：DeepSeek 解析项目模块、建议岗位、技能标签和人数；后端根据技能标签检索候选人画像；按技能匹配、测评结果、学习完成度计算候选人匹配分；生成推荐理由。

输出：项目所需岗位建议、每个岗位的核心技能要求、推荐候选人列表、匹配分、推荐理由。

MVP 限制：使用预置候选人/学生画像数据；不实现完整招聘流程、复杂沟通记录和录用转化漏斗；面试邀约只保留“生成邀约记录”基础能力。

异常处理：项目描述过短时提示补充业务场景；AI 不可用时使用项目关键词规则匹配岗位模板。

## 3.4 管理端基础模块

用户管理：管理员可查看学生、企业 HR、管理员账号列表，支持启用、禁用、重置密码。

岗位技能词典维护：管理员可维护岗位、技能标签、技能等级、岗位技能要求。技能词典作为岗位推荐、差距分析、AI 兜底逻辑的数据基础。

基础数据看板：系统展示用户总数、测评完成数、岗位匹配次数、简历优化次数、模拟面试次数、企业推荐次数等基础指标。

## 3.5 智能客服基础模块

智能客服支持文字问答，回答内容包括平台功能使用说明、常见互联网岗位介绍、学习路径建议、简历和面试准备建议。MVP 阶段不实现语音交互和人工客服流转。对于无法回答的问题，系统提示“建议联系平台管理员”。

### 3.6.2 E-R 关系说明

- 一个用户可对应一个求职画像。
- 一个用户可拥有多条测评结果、推荐记录、学习任务、简历分析记录和面试记录。
- 一个岗位可关联多个技能要求。
- 一个技能可被多个岗位复用。
- 企业 HR 的项目推荐结果复用用户画像、岗位、技能词典和推荐记录表。

# 4 性能需求

## 4.1 时间性能需求

| 场景 | 性能要求 |
|---|---|
| 普通页面加载 | 3 秒内完成主要内容展示 |
| 登录、查询、保存 | 1 秒内返回处理结果 |
| 岗位匹配推荐 | 3 秒内返回推荐列表 |
| AI 职业方向探索 | 15 秒内返回结果 |
| 简历优化 | 20 秒内返回分析结果 |
| 模拟面试单轮问答 | 15 秒内返回追问或评价 |
| 企业项目推荐 | 20 秒内返回岗位和候选人推荐 |

## 4.2 系统开放性需求

岗位技能词典支持后台维护；AI Prompt 模板集中管理；推荐算法权重可配置；后续可扩展真实招聘数据、向量检索、微信通知、语音面试等能力。

## 4.3 界面友好性需求

页面结构清晰，学生端突出当前求职阶段和下一步操作；图表使用 ECharts 展示雷达图、热力图、成长曲线；表单需有必填校验和错误提示；企业端项目推荐页面需突出“输入项目描述 → 生成岗位建议 → 推荐候选人”的演示路径。

## 4.4 系统可用性需求

DeepSeek API 不可用时，核心流程不应完全中断；系统需提供基础错误提示，不展示后端堆栈信息给普通用户；文件上传需限制格式和大小，避免影响系统稳定性。

## 4.5 可管理性需求

管理员可维护用户状态和技能词典；系统记录 AI 调用输入摘要、输出结果和调用状态，便于排查问题；关键业务数据支持分页查询；演示数据可通过后台维护或初始化脚本导入。

# 5 接口需求

## 5.1 用户接口

| 页面 | 主要内容 |
|---|---|
| 登录注册页 | 账号登录、注册入口 |
| 学生首页 | 求职进度、推荐岗位、学习任务 |

## 5.2 软件接口

DeepSeek API：用于职业方向推荐、简历优化、模拟面试、项目需求解析、智能客服；调用方式为 HTTPS REST API；输入为 Prompt、用户画像、测评结果、岗位 JD、项目描述等；输出为 JSON 或文本；异常处理为超时重试 1 次，失败后使用本地规则或缓存兜底。

数据库接口：后端通过 MyBatis-Plus 访问 MySQL，负责用户、岗位、技能、测评、推荐等数据的增删改查。

缓存接口：Redis 用于登录验证码和会话缓存、高频岗位技能数据缓存、AI 调用结果短期缓存。

## 5.3 硬件接口

本系统无专用硬件接口。部署服务器建议配置为 2 核 CPU、4 GB 内存、40 GB 磁盘，并保证可访问 DeepSeek API。

## 5.4 通讯接口

前端与后端通过 HTTP/HTTPS REST API 通信；后端与 DeepSeek API 通过 HTTPS 通信；部署环境建议使用 Nginx 反向代理前端静态资源和后端 API。

# 6 总体设计约束

## 6.1 标准符合性

API 返回结构统一使用 JSON；密码不得明文存储，需使用哈希加密；文件上传需校验类型和大小；用户隐私数据仅用于平台功能分析，不展示给无权限用户。

## 6.2 硬件约束

系统需支持普通云服务器单机部署；不依赖 GPU；AI 推理调用云端 DeepSeek API，不在本地部署大模型。

## 6.3 技术限制

开发周期为 30 天，避免引入过重技术栈；不使用复杂微服务架构，采用单体后端应用；不强依赖向量数据库，优先使用 MySQL 结构化数据和轻量规则实现；企业端候选人推荐使用演示数据，不接入外部招聘平台。

# 7 软件质量特性

## 7.1 可靠性

关键业务接口需进行参数校验；AI 调用需设置超时时间和异常兜底；数据库关键表需设置主键、外键或逻辑关联字段；系统异常时返回友好错误信息。

## 7.2 易用性

学生端流程应按求职路径组织，减少用户理解成本；测评、推荐、差距分析、学习路径之间应有明确下一步入口；企业端亮点功能应能在一个页面完成输入、解析、推荐展示；管理端表格支持搜索和分页。

## 7.3 可维护性

技能词典、岗位模板、Prompt 模板应集中维护；业务模块分层清晰；前端页面按角色和业务模块拆分；关键配置通过配置文件或数据库维护。

## 7.4 安全性

普通用户只能访问自己的画像、测评、简历和面试记录；企业 HR 只能查看用于演示推荐的候选人摘要信息，不展示敏感隐私；管理员操作需登录鉴权；文件上传需防止非法格式和超大文件。

# 9 附录

## 9.1 MVP 核心业务流程

1. 学生注册并填写求职画像。
2. 学生完成职业能力测评。
3. AI 根据画像和测评结果推荐岗位方向。
4. 学生选择目标岗位。
5. 系统生成岗位匹配结果和能力差距报告。
6. 系统生成个性化学习路径。
7. 学生使用简历优化和模拟面试功能准备求职。
8. 企业 HR 输入项目描述，系统生成岗位建议并推荐候选人。

## 9.2 AI 兜底策略

当 DeepSeek API 不可用、超时或返回格式异常时，系统按以下方式降级：职业方向推荐使用测评分数和技能标签进行规则推荐；岗位匹配使用岗位技能词典计算匹配分；差距分析直接对比岗位技能要求和用户技能等级；简历优化返回预置的简历检查清单；模拟面试使用本地题库和评分模板；企业推荐使用项目关键词匹配岗位模板和候选人技能标签。

## 9.3 后续迭代建议

- 引入向量数据库提升岗位、简历和项目描述的语义匹配质量。
- 接入真实招聘网站或企业岗位数据。
- 完善企业端招聘流程，包括邀约、沟通、录用转化统计。
- 增加高校管理后台，支持按专业、年级查看学生就业能力分布。
- 增加语音模拟面试和语音智能客服。
- 增加微信或邮件学习提醒。
