## 1. 功能概述

### 1.1 功能定义

企业 HR 输入项目描述文本（如"开发一个在线商城，包含用户端、商家后台、推荐引擎、订单支付"），系统通过 **DeepSeek 大模型解析项目需求**，自动识别所需岗位和技能要求，然后从学生人才库中**多维度评分匹配候选人**，输出岗位建议+候选人推荐列表+推荐理由。

### 1.2 功能定位

| 维度 | 说明 |
|------|------|
| **优先级** | A 级（核心演示功能） |
| **用户角色** | 企业 HR |
| **输入** | 项目描述文本（≥ 20 字）、可选筛选条件（学历、城市、技能偏好） |
| **AI 模型** | DeepSeek Chat API（项目需求解析）+ 规则引擎（候选人匹配评分） |
| **输出** | 项目解析摘要、建议岗位列表、岗位技能要求、候选人推荐列表（含多维度评分和推荐理由） |
| **MVP 约束** | 使用系统内已注册的学生数据；不接入外部招聘平台 |

### 1.3 与现有代码的关系

当前代码（`EnterpriseServiceImpl.recommend()`）为 **FALLBACK 模拟模式**，使用随机数生成评分。本计划将其升级为**真实 AI 解析 + 多维度规则评分**。

```
└───────┬───────────────┘
        │
        ▼
┌───────────────────────┐
│ ③ 候选人池召回        │  ← 规则引擎 + DB查询
│ → 从 user 表查学生     │

│ → 查学习完成度         │
└───────┬───────────────┘
        │
        ▼

│ ④ 多维度加权评分       │  ← 规则引擎（详见第3章）
│ → 技能匹配 50%         │
│ → 测评适配 25%         │
│ → 学习完成 15%         │
│ → 城市/学历 10%        │
└───────┬───────────────┘
        │
        ▼
┌───────────────────────┐
│ ⑤ DeepSeek 生成推荐理由 │ ← AI 调用（同步）

└───────┬───────────────┘

        ▼

│ → 写入 recommendation   │
│   _record 表            │
│ → 返回结构化 JSON       │
└───────────────────────┘
```

**性能目标**：全流程 ≤ 20 秒（PRD 4.1 要求）

## 3. 评分机制

### 3.1 总体评分模型

```
总分 = W1 × 技能匹配分 + W2 × 测评适配分 + W3 × 学习完成分 + W4 × 基础匹配分
```

| 权重项 | 代号 | 权重 | 说明 | 来源 |
|--------|------|------|------|------|
| 技能匹配 | W1 | **50%** | 学生技能与岗位要求的重叠度 | 规则计算 |
| 测评适配 | W2 | **25%** | 学生能力测评与岗位类型的适配 | 规则计算 |
| 学习完成 | W3 | **15%** | 学生相关技能的学习任务完成度 | 规则计算 |
| 基础匹配 | W4 | **10%** | 学历、城市、薪资期望匹配 | 规则计算 |

> 权重与 PRD 3.2.4 节保持一致，后续可通过配置文件热调整。

### 3.2 各维度详细计算

#### 3.2.1 技能匹配分（权重 50%，满分 100）

**数据来源**：
- 候选人：`career_profile.skill_tags`（JSON 数组）
- 岗位要求：DeepSeek 解析出的技能列表（映射到 `skill` 表）或 `job_skill_requirement` 表

**计算逻辑**：

```
1. 获取岗位要求的所有技能及要求等级（Level_R）
2. 获取学生已掌握的技能及等级（Level_S）
3. 对每个岗位要求技能：
   - 如果学生无该技能 → 贡献 0
   - Level_S >= Level_R → 贡献 Weight × 1.0（完全满足）
   - Level_S = Level_R - 1 → 贡献 Weight × 0.6（基本满足）
   - Level_S < Level_R - 1 → 贡献 Weight × 0.3（有差距）
4. 技能匹配分 = Σ(单项贡献) / Σ(技能权重) × 100
```

**技能等级映射（数值化）**：

| 等级 | 数值 | 说明 |
|------|------|------|
| 了解 | 1 | BASIC |
| 掌握 | 2 | MASTER |
| 熟练 | 3 | PROFICIENT |
| 精通 | 4 | EXPERT |

**示例**：
```
岗位要求: Java(精通, 权重0.4) + Spring Boot(熟练, 权重0.3) + Redis(掌握, 权重0.3)
学生技能: Java(熟练, Level=3) + Spring Boot(掌握, Level=2) + Redis(了解, Level=1)

Java:      Level_S=3 < Level_R=4, 差1级 → 0.4 × 0.6 = 0.24
Spring:    Level_S=2 < Level_R=3, 差1级 → 0.3 × 0.6 = 0.18
Redis:     Level_S=1 < Level_R=2, 差1级 → 0.3 × 0.6 = 0.18
总分 = (0.24+0.18+0.18) / 1.0 × 100 = 60
```

#### 3.2.2 测评适配分（权重 25%，满分 100）

**数据来源**：`assessment_result` 表（学生最近一次测评）

**岗位类型 → 测评维度权重映射**：

| 岗位类型 | 编程基础 | 逻辑推理 | 产品思维 | 技术理解 | 沟通协作 |
|----------|----------|----------|----------|----------|----------|
| 后端开发 | 35% | 25% | 10% | 20% | 10% |
| 前端开发 | 25% | 20% | 20% | 20% | 15% |
| 产品经理 | 10% | 20% | 40% | 15% | 15% |
| 数据分析 | 20% | 30% | 15% | 25% | 10% |
| 测试工程师 | 20% | 25% | 10% | 25% | 20% |
| 运营岗 | 10% | 15% | 30% | 10% | 35% |
| **默认** | 20% | 20% | 20% | 20% | 20% |

**计算公式**：

```
测评适配分 = Σ(维度得分 × 岗位维度权重) / 100
```

其中各维度得分来自 `assessment_result` 表的 `programming_score`, `logic_score`, `product_score`, `tech_score`, `communication_score`（每维度满分 100）。

**示例（后端开发）**：
```
编程=85, 逻辑=70, 产品=60, 技术=80, 沟通=75
测评分 = 85×0.35 + 70×0.25 + 60×0.10 + 80×0.20 + 75×0.10 = 29.75+17.5+6+16+7.5 = 76.75
```

#### 3.2.3 学习完成分（权重 15%，满分 100）

**数据来源**：`learning_task` 表

**计算逻辑**：

```
1. 查询学生在推荐岗位相关技能下的学习任务
2. 统计：
   - 总任务数（T_total）
   - 已完成任务数（T_done）
   - 进行中任务数（T_doing）
   - 任务阶段权重：基础夯实(0.15) / 框架掌握(0.3) / 项目实战(0.35) / 面试冲刺(0.2)
3. 学习完成分 = [T_done + T_doing × 0.5] / T_total × 100
   （进行中任务按 50% 计入）

若无学习任务，默认 50 分（中等水平）。
```

#### 3.2.4 基础匹配分（权重 10%，满分 100）

**数据来源**：`career_profile` 表

| 子项 | 权重 | 匹配规则 |
|------|------|----------|
| 学历匹配 | 30% | 学历一致 100，相邻等级 70，差距大 30 |
| 城市匹配 | 30% | 同城 100，同省 70，其他 30 |
| 薪资匹配 | 20% | 岗位薪资在期望区间内 100，差距 <20% 70，其他 30 |
| 专业相关 | 20% | 专业与岗位方向的关联度（规则映射表） |

**学历等级映射**：

| 学历 | 数值 |
|------|------|
| 博士 | 5 |
| 硕士 | 4 |
| 本科 | 3 |
| 大专 | 2 |
| 其他 | 1 |

**专业-岗位关联度映射表**（部分示例）：

| 专业关键字 | 关联岗位方向 | 关联度 |
|------------|--------------|--------|
| 计算机科学/软件工程/人工智能 | 后端/前端/算法/数据 | 100 |
| 电子信息/通信工程 | 后端/硬件/测试 | 80 |
| 数学/统计 | 数据/算法/后端 | 75 |
| 产品设计/交互设计 | 前端/产品 | 70 |
| 其他专业 | 通用 | 40 |

---

### 3.3 综合评分与等级

```
综合分 = 0.50 × 技能匹配 + 0.25 × 测评适配 + 0.15 × 学习完成 + 0.10 × 基础匹配
```

**推荐等级**：

| 综合分 | 等级 | 建议操作 |
|--------|------|----------|
| ≥ 85 | ⭐⭐⭐⭐⭐ 非常匹配 | 优先沟通 |
| 70-84 | ⭐⭐⭐⭐ 比较匹配 | 列入候选 |
| 60-69 | ⭐⭐⭐ 一般匹配 | 可考虑 |
| 45-59 | ⭐⭐ 部分匹配 | 需培养 |
| < 45 | ⭐ 不太匹配 | 不建议 |

## 4. AI 集成方案

### 4.1 DeepSeek 调用的两个环节

| 环节 | 步骤 | 调用方式 | 超时 | 说明 |
|------|------|----------|------|------|
| **项目需求解析** | 步骤 ① | RestTemplate 同步 | 15 秒 | 解析项目描述→岗位+技能建议 |
| **候选推荐理由** | 步骤 ⑤ | RestTemplate 同步 | 15 秒 | 为 Top 3-5 候选人生成个性化推荐理由 |

### 4.2 项目需求解析 Prompt 设计

**Prompt 模板**（基础层，存储为文件 `project_parse.txt`，不可后台编辑）：

```
你是一位资深技术招聘顾问。请根据以下项目需求描述，分析项目所需的技术岗位和技能要求。

项目需求：{{projectDescription}}

请以 JSON 格式返回分析结果（严格保持以下结构）：
{
  "projectSummary": "项目一句话概述",
  "positions": [
    {
      "positionTitle": "岗位名称（如 后端开发工程师）",
      "skillRequirements": [
        {"skillName": "技能名称", "requiredLevel": "精通/熟练/掌握/了解"}
      ],
      "headcount": 需求人数建议（数字）
    }
  ]
}

分析要求：
1. 输出 2-5 个岗位建议
2. 每个岗位列出 3-8 个核心技能
3. 技能等级使用：精通/熟练/掌握/了解
4. 技能名称使用行业通用名称（如 Java、Spring Boot、MySQL 等）
5. 如果项目描述信息不足，请根据行业经验合理推断
```

**DeepSeek 调用参数**：

```json
{
  "model": "deepseek-chat",
  "messages": [{
    "role": "user",
    "content": "（组装后的 Prompt）"
  }],
  "temperature": 0.3,
  "max_tokens": 2000,
  "response_format": { "type": "json_object" }
}
```

> 使用 `response_format: json_object` 确保 AI 返回合法 JSON，降低解析失败率。

### 4.3 候选推荐理由 Prompt 设计

**Prompt 模板**（业务层，存储到 `prompt_template` 表，可后台编辑）：

```
你是招聘辅助助手。根据以下信息，为候选人生成简洁的推荐理由。

岗位需求技能：{{requiredSkills}}
候选人信息：
- 已掌握技能：{{candidateSkills}}
- 能力测评分：编程{{progScore}}、逻辑{{logicScore}}、产品{{prodScore}}、技术{{techScore}}、沟通{{commScore}}
- 匹配分：{{matchScore}}
- 优势技能：{{strengthSkills}}
- 技能缺口：{{gapSkills}}

请生成 1-2 句话的推荐理由，突出候选人的优势和不足，字数 50-80 字。
```

### 4.4 后续增强（第二阶段，向量检索）

按技术路线图第二阶段规划，可引入 Embedding 向量检索增强候选人召回：

1. 对候选人 `career_profile` 摘要生成 Embedding
2. 对岗位 JD 文本生成 Embedding
3. MySQL JSON 存储向量 + 内存余弦相似度重排
4. 语义级匹配提升推荐精度

此项在 MVP 阶段不做，详见决策 3。

## 5. 数据库设计

### 5.1 现有表（无需修改，直接复用）

| 表名 | 用途 |
|------|------|
| `user` | 候选人基础信息（角色=STUDENT） |
| `career_profile` | 求职画像（技能标签、学历、城市、期望薪资） |
| `assessment_result` | 能力测评结果（五维度分数） |

| `job_skill_requirement` | 岗位-技能关联 |
| `learning_task` | 学习任务（完成进度） |
| `recommendation_record` | 推荐记录（存储输入+结果JSON） |

### 5.2 recommendation_record 表使用规范

| 字段 | 赋值说明 |
|------|----------|
| `user_id` | HR 用户 ID（谁发起的推荐） |
| `type` | 固定为 `ENTERPRISE_REC` |
| `input_text` | HR 输入的项目描述（原始文本） |
| `result_json` | 完整推荐结果 JSON（`EnterpriseRecommendResponse` 序列化） |
| `source` | `AI`（AI调用成功） / `FALLBACK`（AI不可用兜底） / `HYBRID`（部分AI+部分规则） |
| `created_at` | 调用时间 |

### 5.3 推荐历史查询

```sql
-- 获取当前 HR 的推荐历史
SELECT id, type, source, created_at, 
       LEFT(input_text, 100) AS input_preview
FROM recommendation_record
WHERE user_id = #{hrUserId}
  AND type = 'ENTERPRISE_REC'
ORDER BY created_at DESC
LIMIT #{limit} OFFSET #{offset}
```

## 8. 兜底策略

### 8.1 兜底触发条件

| 条件 | 触发 |
|------|------|
| DeepSeek API 调用超时（> 15s） | 降级 |
| DeepSeek API 返回 4xx/5xx | 降级 |
| DeepSeek 返回 JSON 格式异常 | 降级 |
| Resilience4j 熔断器开启 | 降级 |
| 网络不可达 | 降级 |

### 8.2 兜底处理方案

| 环节 | 正常模式 | 兜底模式 |
|------|----------|----------|
| 项目需求解析 | DeepSeek 解析 → 岗位+技能建议 | 从项目描述中提取关键词（正则/NLP分词），匹配预置岗位模板和技能词典 |
| 候选人匹配评分 | 规则引擎四维评分（仍需执行） | 规则引擎四维评分（不需要变更，评分计算不依赖 AI） |
| 候选推荐理由 | DeepSeek 生成个性化理由 | 使用模板生成："该候选人在[技能1]、[技能2]方面与岗位要求较匹配，建议关注[技能缺口]方面的培养" |
| 结果来源标记 | `source = "AI"` | `source = "FALLBACK"` |

### 8.3 兜底关键词匹配法

```
1. 对项目描述进行中文分词
2. 提取技术关键词（Java/Spring/MySQL/Redis/Vue...）
3. 查技能词典 → 反向查常见岗位模板
4. 返回预设岗位+技能组合
5. 候选推荐理由使用模板拼接
```

### 8.4 模拟数据兜底（已有，保留）

当数据库中学生候选人为空时（演示环境），使用 `Math.random()` 生成模拟数据，但需要在本计划中**移除随机数**，改为基于真实学生画像的可解释评分。

## 附录 A：评分引擎核心类设计（参考）

```java
/**
 * 候选人评分引擎
 * 职责：纯计算层，不依赖 AI，确保兜底模式下评分依然准确
 */
@Service
public class CandidateScoreServiceImpl implements CandidateScoreService {

    private final CareerProfileMapper profileMapper;
    private final AssessmentResultMapper assessmentMapper;
    private final LearningTaskMapper learningTaskMapper;
    private final RecommendConfig config;

    /**
     * 计算候选人综合匹配分
     */
    public CandidateScore compute(Long userId, List<JobSkillRequirement> jobSkills) {
        CareerProfile profile = getProfile(userId);
        AssessmentResult assessment = getLatestAssessment(userId);
        List<LearningTask> tasks = getLearningTasks(userId, jobSkills);

        double skillScore = calcSkillMatch(profile.getSkillTags(), jobSkills);
        double assessScore = calcAssessmentMatch(assessment, jobSkills);
        double learnScore = calcLearningProgress(tasks);
        double basicScore = calcBasicMatch(profile, filters);

        double total = config.getWeightSkill() * skillScore
                     + config.getWeightAssessment() * assessScore
                     + config.getWeightLearning() * learnScore
                     + config.getWeightBasic() * basicScore;

        return CandidateScore.builder()
                .userId(userId)
                .totalScore(total)
                .skillScore(skillScore)
                .assessmentScore(assessScore)
                .learningScore(learnScore)
                .basicScore(basicScore)
                .matchedSkills(getMatched(profile, jobSkills))
                .gapSkills(getGaps(profile, jobSkills))
                .matchLevel(getLevel(total))
                .build();
    }
}
```
