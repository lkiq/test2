## 用户需求

能力测评模块和模拟面试模块共需优化三个问题：

### 问题1：测评结果报告 - 优势/薄弱维度无针对性

当前后端 `AssessmentServiceImpl.buildReport()` 仅根据固定阈值（>=85分优势、<55分薄弱）简单拼接维度英文名，若无相关维度则显示"无显著优势维度"等固定文案。用户希望根据测试结果，由AI生成有针对性的优势维度分析和薄弱维度改进建议。

### 问题2：答题界面 - 缺少计时和交互按钮

当前 `AssessmentView.vue` 答题界面仅含"上一题/下一题"按钮，缺少：

- 倒计时功能（建议25分钟），时间到自动提交
- "退出测评"按钮及确认对话框
- "提交测评"按钮及确认对话框，含未答题提醒
- 题目导航显示已答/未答状态

### 问题3：面试兜底评分算法化（新增）

当前 `InterviewServiceImpl.localEvaluation()` 使用 `75.0 + random() * 20` 随机评分+硬编码维度分和评语，缺乏参考价值。需设计轻量本地降级算法，基于：

- **回答字数**：回答越详实得分越高
- **关键词匹配密度**：为每道本地面试题定义关键词列表，计算命中率
- **作答耗时**：记录每题用时，过短说明敷衍，过长说明卡顿
- **分维度评分**：逻辑(logic)、专业性(professionalism)、沟通(communication)、适应性(adaptability)、岗位匹配(jobFit)分别评分
- **自动生成评语**：基于分析结果动态生成亮点和改进建议

## 技术栈

- **前端**：Vue 3 + TypeScript + Pinia + Element Plus
- **后端**：Spring Boot 3 + Java 17 + MyBatis-Plus
- **AI服务**：DeepSeek API（项目已有封装：`DeepSeekService`）
- **缓存**：Redis（模拟面试Session存储）

## 实现方案

### 方案1：AI生成测评分析结论

**变更文件**：`AssessmentServiceImpl.java`、`AssessmentReportResponse.java`

在 `submitAssessment()` 计算完各维度得分后，调用 `DeepSeekService.callAPIWithCache()` 生成AI分析：

1. **提示词设计**：systemPrompt="你是一位专业的职业能力测评分析师"，传入五维得分（编程能力/逻辑推理/产品思维/技术素养/沟通表达）及等级
2. **AI输出格式**：JSON格式，包含 `strengthAnalysis`（优势分析）、`weaknessAnalysis`（薄弱分析）、`suggestions`（学习建议）
3. **数据存储**：AI分析结果存入 `AssessmentResult.resultJson` 字段（该字段已存在）
4. **响应扩展**：`AssessmentReportResponse` 新增 `aiAnalysis` 字段（Map类型），前端展示时优先显示AI结论，原有的 `strengths`/`weaknesses` 保留作为兜底
5. **降级**：AI不可用时，回退到原有阈值判断逻辑

### 方案2：答题界面交互增强

**变更文件**：`assessment.ts`（Store）、`AssessmentView.vue`

**Store扩展**：

- 新增 `startTime`（开始时间戳）、`timeLimit`（限时秒数，默认1500秒=25分钟）
- 新增 `remainingTime`（剩余秒数，响应式）
- 新增 `timerInterval`（定时器引用）

**UI增强**：

- 答题阶段顶部添加倒计时进度条（显示剩余 `MM:SS` 格式），使用Element Plus `ElProgress`，颜色随剩余时间变化（绿色>黄色>红色）
- 添加"退出测评"按钮，使用 `ElMessageBox.confirm` 确认后返回首页
- 添加"提交测评"按钮（主色高亮），点击时检查未答题数量并弹窗提示
- 题目导航区域显示已答/未答状态小圆点

**边界处理**：

- 倒计时归零时自动调用提交逻辑
- 组件卸载时使用 `onUnmounted` 清除定时器
- 退出时保留已答数据防误触丢失

### 方案3：面试本地降级评分算法

**变更文件**：`InterviewServiceImpl.java`

#### 3.1 扩展本地题库，增加关键词定义

```java
// 为5道本地面试题分别定义领域关键词
private static final Map<Integer, List<String>> QUESTION_KEYWORDS = Map.of(
    0, List.of("项目", "经验", "技术", "能力", "方向", "专业", "学习", "团队"),
    1, List.of("挑战", "方案", "解决", "技术", "团队", "成果", "难点", "优化"),
    2, List.of("封装", "继承", "多态", "设计", "模式", "接口", "复用", "耦合"),
    3, List.of("排查", "日志", "监控", "性能", "负载", "缓存", "数据库", "优化"),
    4, List.of("规划", "目标", "发展", "技术", "管理", "方向", "成长", "学习")
);
```

#### 3.2 记录作答耗时

在 `submitAnswer()` 中，从session获取当前题目的开始时间(`questionStartTime`)，计算耗时（秒）并存入答案记录；在返回 `InterviewSession` 时携带当前题目的开始时间戳。

session中新增字段：

- `questionStartTime`：当前题目开始展示时的时间戳（System.currentTimeMillis）

#### 3.3 实现加权评分算法

修改 `localEvaluation()` 的签名，接收session参数，获取答案列表进行分析：

**评分维度权重**（每题满分100分 -> 加权到各维度）：

1. **回答长度评分**（权重30%）

- 统计中文字符数（去除空白字符）
- >=150字：满分30；100-149字：22分；50-99字：15分；20-49字：8分；<20字：3分

2. **关键词匹配密度**（权重40%）

- 每道题有独立的领域关键词列表
- 关键词命中率 = 命中的关键词数 / 总关键词数
- 匹配分 = 命中率 * 40

3. **作答耗时合理性**（权重30%）

- 统计每题作答耗时（毫秒）
- 合理区间：30秒-180秒
- <10秒（敷衍）：5分；10-30秒（偏短）：15分；30-180秒（合理）：30分；>180秒（过长）：20分

**分维度汇总**：

- 每题按题目类型映射到不同维度权重，累计各维度总分
- 第0题(自我介绍)->communication权重高; 第1题(项目经验)->professionalism+logic; 第2题(技术基础)->professionalism; 第3题(情景模拟)->adaptability+logic; 第4题(职业规划)->jobFit+communication

**动态评语生成**：

- 按各维度平均分生成分级评语：>=85"优秀"，70-84"良好"，55-69"一般"，<55"待提升"
- 取最高分维度生成亮点，最低分维度生成改进建议
- 综合评语根据整体表现组合

### 架构：双路径分析

```
submitAssessment()
    ├── 1. 计算维度得分（原有逻辑不变）
    ├── 2. FallbackRuleLibrary.buildFallbackAnalysis(scores) → fallbackAnalysis（始终生成）
    ├── 3. DeepSeekService.callAPIWithCache(...) → aiAnalysis（仅 API 可用时）
    │       └── 失败/不可用 → 静默降级，log.warn 记录
    ├── 4. aiAnalysis + fallbackAnalysis 一起存入 resultJson（JSON 序列化）
    └── 5. buildReport() → AssessmentReportResponse（前端取 aiAnalysis || fallbackAnalysis）
```

### AI System Prompt 核心约束

- 输出严格 JSON，三个模块：strengths / weaknesses / overallSummary
- strengths 每项含 techApplications + compensationStrategies
- weaknesses 每项含 rootCause + tieredPlan（beginner/intermediate/advanced）
- 禁止空泛评价，所有结论基于分数事实
- tasks 必须是可执行动作（动词+技术名+工具名）
- 技术方案与 Java/Vue/Spring Boot 技术栈绑定

### FallbackRuleLibrary 规则规模

| 规则集 | 维度数 | 条目数 | 说明 |
|--------|--------|--------|------|
| STRENGTH_APPS | 5 | 3-4 条 TechApplication/维 | 技术落地场景 + 工具 + 产出 |
| COMPENSATIONS | 5 | 2 条 CompensationStrategy/维 | 以长补短（仅当薄弱维度真实存在） |
| WEAKNESS_PLANS | 5 | 3 层 TierLevel × 4-5 个 Task/层 | 入门→进阶→实战，含前置条件+验收标准+周投入时间 |

### 优势/薄弱判定规则（替代原 85/55 一刀切）

**优势判定**：
- ≥ 85 分 → 显著优势
- 70-84 分且为最高分 → 潜在优势
- 全部 < 85 时，取最高分作为唯一优势（即便只有 60 分）

**薄弱判定**：
- < 55 分 → 绝对薄弱
- 55-69 分但与最高维度差距 ≥ 25 分 → 相对薄弱

### 设计决策确认

1. **分层方案粒度**：保留入门/进阶/实战三层
2. **以长补短策略**：预置规则库自动匹配（仅显示指向真实薄弱维度的策略）
3. **兜底规则库维护**：硬编码在 `FallbackRuleLibrary.java`（单独抽类管理，不混入 Service）
4. **AI 缓存策略**：TTL 1 小时，key 模式 `assessment:ai:{userId}:{timestamp}`
5. **薄弱判定**：支持绝对薄弱（<55）+ 相对薄弱（比最高分低 ≥25）
